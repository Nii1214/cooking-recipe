# ADR 001: Next.js (App Router)

## 背景
このアプリはクリーンアーキテクチャを前提に設計している。
ルーティング・サーバーサイド処理・API構築を統一されたフレームワークで扱いたい。
クローズド環境で重要なデータを扱うため、フロントだけでなくバックエンドも統合されたフレームワークが必要。

## 決定内容
Next.js の App Router を採用する。

## 結果・利点
- ページ単位で Server Action を定義でき、UseCase層にロジックを委譲可能
- クリーンアーキテクチャとの親和性が高い
- Vercel との統合で CI/CD 自動化が容易
- 将来的にサーバーサイドレンダリングや ISR も対応可能

## 考慮点・トレードオフ
- 学習コストが Pages Router より高い
- App Router の新しい API に依存するため、将来の仕様変更に注意
- 小規模個人アプリではオーバーアーキテクチャになり得る

## 他フレームワークとの比較

### React 単体
- フロントのみのライブラリでサーバサイド処理は別途設計が必要
- Clean Architecture を守るにはサーバ処理の層も自分で設計する必要がある
- 手間が増えるため見送り

### Nuxt.js
- Vue エコシステム向け、SSR 対応あり
- TypeScript 対応は React / Next.js よりやや劣る
- 再利用性・エコシステムを考慮して Next.js を採用
