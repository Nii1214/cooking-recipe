# Supabase

## 背景
- 家族単位でデータを閉じたい  
- 認証・ストレージ・データベースを統合したい  
- RLS(Row Level Security)によるマルチテナント設計を実装したい

## 決定内容
Supabase を採用する

## 結果・利点
- Postgres + RLS で家族単位の分離が可能
- 認証・ストレージ・DBを統合で利用可能
- 簡単に永続化戦略を実装できる
- 将来的な拡張や他クラウド移行が容易な抽象化も可能

## 考慮点・トレードオフ
- RLS だけでは安全性保証できないので、ドメイン層でも整合性チェックを二重に行う
- ストレージ容量・課金設計に注意
